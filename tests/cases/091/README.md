# Case 091

Buried-slab fixture for the `BCcleanup` regression harness
(`tests/regression/bc_cleanup/`), backlog item ┬¦6.6. Copied from
`tests/cases/090/` (see `tests/cases/090/README.md` for its own lineage)
with the geometry replaced by a flat box covering the *entire* floor of
the domain (`xlen = ylen = 64`) to a height of 2 grid cells
(`dz = 1`, so `z in [0, 2]`).

Purpose: exercise the base-state fallback added to `diagfld` in
`src/modthermodynamics.f90`, which substitutes base-state profiles
wherever a z-slab has zero fluid cells (`IIcs(k) == 0`). With this
geometry, the two lowest slabs (`kb` and `kb+1`, i.e. k=1 and k=2 on the
c-grid) are fully solid ŌĆö `avexy_ibm` would previously return the
`-999.` sentinel there.

Differences from case 090:

- `iexpnr = 091` (was `090`).
- `stl_file = geom.091.STL` (was `geom.101.STL`): replaced with a
  programmatically generated, watertight 12-triangle box (via
  `trimesh.creation.box`) spanning `x in [0,64]`, `y in [0,64]`,
  `z in [0,2]` ŌĆö i.e. the entire domain footprint, 2 grid cells tall.
  Outward-facing normals verified at generation time.
- `&WALLS` counts (`nfcts`, `nsolpts_*`, `nbndpts_*`, `nfctsecs_*`)
  regenerated by the Python IBM preprocessor for the new geometry
  (`nfcts = 12`; only the top face's 2 triangles are matched by
  `facet_sections_c` ŌĆö the bottom face and the 4 side faces coincide
  with the domain floor / periodic x-y boundaries and are geometrically
  unexposed, hence `facets_unused.091` lists 10 of the 12 facets).
- `Tfacinit.inp.091` regenerated to match the new `nfcts = 12` (was
  stale at 320 rows, copied from case 090's building geometry).
- `&INPS` `zSb`/`zSe` moved from `1` to `3` (physical z, metres) so the
  scalar line source sits above the buried slab instead of inside it;
  `scalarsourcel.inp.1.091` regenerated to match.
- All `solid_*.txt`, `fluid_boundary_*.txt`, `facet_sections_*.txt`,
  `facets.inp.091`, `facetarea.inp.091`, `facets_unused.091` regenerated
  from the new STL via `UDPrep(...).ibm.run_all(backend="f2py")`.

Regeneration was done on Windows following
`docs/udales-preprocessing-windows.md`, using the repo's prebuilt
f2py extension modules under `tools/python/udprep/*.pyd` (rebuilt from
current source with the local `gfortran`/`gcc`/`cmake` toolchain, since
the checked-in `.pyd` binaries predated the current `ibm_preproc_f2py`
Fortran interface).

Static verification: every one of `solid_c.txt`, `solid_u.txt`,
`solid_v.txt` has exactly `itot*jtot = 4096` unique `(i,j)` pairs at
both `k=1` and `k=2`, and no entries at `k>=3` ŌĆö i.e. the two lowest
slabs are fully solid and nothing above is.

Not verified: the solver was not run in this environment (no Fortran
build toolchain for `u-dales` itself, only for the Python preprocessing
f2py extensions). See `tests/regression/bc_cleanup/run_test.py
--assert-only 091` for the intended runtime check (run completes,
startup log contains "Base state:", no `-999.`/NaN in dumped fields).
